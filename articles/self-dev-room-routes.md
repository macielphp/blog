# How the **Self Development Room** Routes Were Designed

**This article is based on the exact implementation found in commit:**
[https://github.com/macielphp/self-development-room/commit/696c59ea96c75f517ba4808f7c9bcd249e7329b4](https://github.com/macielphp/self-development-room/commit/696c59ea96c75f517ba4808f7c9bcd249e7329b4)

This article explains the **architecture, principles, and technical decisions** behind the routing system of the **Self Development Room** project, covering both the **back-end (API)** and the **front-end**, with a strong focus on **security, scalability, and mental clarity**.

---

## 🎯 Routing Architecture Goals

The project was designed to serve **two clearly distinct worlds**:

* **Administration** (content, languages, seasons, lessons, questions)
* **End users** (learning flow, progress, authentication)

Because of this, the routing architecture was designed around:

* Clear separation of responsibilities
* Layered security
* Future scalability
* Readability and maintainability

---

## 🧱 Core Principle: Domain Separation

From the very beginning, the following rule guided the design:

> **Admin and User must never share the same protected route space.**

This prevents:

* Permission leaks
* Ambiguous authentication states
* Unnecessary coupling

As a result, the back-end was divided into **three major blocks**:

1. Public routes (`/auth`)
2. Administrative routes (`/admin/*`)
3. Product routes (authenticated users)

---

## 🔓 Public Routes – `/auth`

These routes **do not require authentication** and exist exclusively for **entering the system**.

### Examples:

* `POST /auth/register`
* `POST /auth/login`
* `POST /auth/google`
* `POST /auth/resetpassword`
* `POST /auth/resetpassword/confirm`
* `POST /auth/refresh`

### Why isolate them?

* Avoid unnecessary middleware
* Reduce attack surface
* Enable aggressive rate limiting

These routes **never expose system data** — they only generate **temporary credentials**.

---

## 🛡️ Authentication with Access Token + Refresh Token

The system uses:

* **Access Token (short-lived JWT)** → sent via the `Authorization` header (15 minutes)
* **Refresh Token (long-lived JWT)** → stored in an **httpOnly cookie** (7 days)

### Why this model?

* Protection against XSS attacks on token storage
* Token revocation capability
* Silent session renewal
* Industry-standard pattern

Important clarification:

> **The refresh token is never provided by third parties. It is always generated by the back-end.**

### Token Flow:

```javascript
// Login → Set refresh token in httpOnly cookie
res.cookie('refreshToken', refreshToken, {
    httpOnly: true,
    secure: isProd,
    sameSite: isProd ? 'none' : 'lax',
    maxAge: 7 * 24 * 60 * 60 * 1000
});

// Return access token in response body
res.json({ accessToken, user });
```

---

## 👤 Authenticated User Routes – Product Layer

After authentication, all user routes pass through:

```js
authUser → rate limiter → product routes
```

### Example grouping:

```js
app.use(
  '/',
  authUser,
  userApiLimiter,
  userProfileRoutes,
  userLanguageRoutes,
  userSeasonRoutes,
  userLessonRoutes,
  userActivityRoutes
);
```

### Benefits of this approach

* Middleware applied **only once**
* Less repetition
* Cleaner code
* Guaranteed user identity

---

## 🧠 `authUser` Middleware

Responsible for:

* Validating the Access Token
* Ensuring token expiration
* Injecting the user into `req`

```js
const authUser = (req, res, next) => {
    const authHeader = req.headers.authorization;

    if (!authHeader || !authHeader.startsWith('Bearer ')) {
        return res.status(401).json({ 
            message: 'Authentication token missing' 
        });
    }

    const token = authHeader.split(' ')[1];

    try {
        const decoded = jwt.verify(token, process.env.JWT_ACCESS_SECRET);

        req.user = {
            id: decoded.id,
            email: decoded.email,
            role: decoded.role
        };
        
        next();
    } catch (err) {
        return res.status(401).json({ 
            message: 'Invalid or expired token' 
        });
    }
};
```

This allows any controller to:

* Know who the user is
* Apply business rules
* Avoid redundant database queries

---

## 🧑‍💼 Administrative Routes – `/admin/*`

Administrative routes are **fully isolated**:

* Dedicated authentication
* Dedicated JWT secret (`JWT_ADMIN_SECRET`)
* Dedicated middleware (`authAdmin`)

### Example:

```js
app.use('/admin/languages', authAdmin, adminLanguagesRoutes);
app.use('/admin/seasons', authAdmin, adminSeasonsRoutes);
app.use('/admin/lessons', authAdmin, adminLessonRoutes);
```

### Why this matters

* Admins require elevated privileges
* Different token verification logic
* Prevents privilege escalation attacks
* Clear separation of concerns

This design prevents critical security flaws where a user token could potentially be used to access admin routes.

## 🚦 Rate Limiting with Redis

The system implements **three layers of rate limiting**:

### 1. Admin Login Rate Limiting

```js
// 3 attempts per 15 minutes
const adminLoginLimiter = rateLimit({
    store: new RedisStore({
        sendCommand: (...args) => redisClient.sendCommand(args)
    }),
    windowMs: 15 * 60 * 1000,
    max: 3,
    message: { error: 'Too many login attempts. Try again later.' }
});
```

### 2. User Login Rate Limiting

```js
// 5 attempts per 15 minutes
const userLoginLimiter = rateLimit({
    store: new RedisStore({
        sendCommand: (...args) => redisClient.sendCommand(args)
    }),
    windowMs: 15 * 60 * 1000,
    max: 5,
    message: { error: 'Too many login attempts. Try again later.' }
});
```

### 3. General API Rate Limiting

```js
// 60 requests per minute for authenticated users
const userApiLimiter = rateLimit({
    store: new RedisStore({
        sendCommand: (...args) => redisClient.sendCommand(args)
    }),
    windowMs: 1 * 60 * 1000,
    max: 60
});
```

### Why Redis?

* Protection against brute-force attacks
* Protection against email enumeration
* High performance (in-memory storage)
* Distributed rate limiting across multiple servers

**Important**: Current implementation uses IP-based rate limiting. For enhanced security, consider implementing **user-based rate limiting** after authentication to prevent distributed attacks from multiple IPs targeting the same account.

## 🔑 Password Reset: A Special Case

Password reset **does not use JWT**.

### Why?

* Single-use token
* Short expiration time (15 minutes)
* Non-reusable
* Manually invalidatable
* No need for cryptographic verification overhead

### The flow:

```js
// 1. Generate random token
const resetToken = crypto.randomBytes(32).toString('hex');

// 2. Hash token before storing
const hashedToken = crypto
    .createHash('sha256')
    .update(resetToken)
    .digest('hex');

// 3. Store hashed token with expiration
await pool.query(`
    UPDATE users
    SET reset_password_token = $1,
        reset_password_expires = $2
    WHERE id = $3
`, [hashedToken, expires, userId]);

// 4. Send unhashed token via email
const resetLink = `${FRONTEND_URL}/resetpassword/confirm?token=${resetToken}`;

// 5. Verify and invalidate after use
await pool.query(`
    UPDATE users
    SET password = $1,
        reset_password_token = NULL,
        reset_password_expires = NULL
    WHERE id = $2
`, [hashedPassword, userId]);
```

This token **does not replace JWTs** — it solves a different problem: secure, one-time operations.

## 🔄 Token Refresh Flow

When the access token expires, the client can silently refresh it:

```js
const refreshTokenController = (req, res) => {
    const refreshToken = req.cookies.refreshToken;

    if (!refreshToken) {
        return res.status(401).json({ message: 'No refresh token' });
    }

    try {
        const decoded = jwt.verify(
            refreshToken, 
            process.env.JWT_REFRESH_SECRET
        );

        const newAccessToken = generateAccessToken({
            id: decoded.id,
            email: decoded.email,
            role: 'user'
        });

        res.json({ accessToken: newAccessToken });
    } catch (error) {
        return res.status(401).json({ 
            message: 'Invalid refresh token' 
        });
    }
};
```

### Current Limitations & Future Improvements

**What's implemented:**
* Silent token renewal
* Separate refresh endpoint

**What's missing (recommended for production):**
* **Refresh token rotation**: Issue new refresh token with each refresh
* **Token revocation/blacklist**: Track invalidated tokens in Redis
* **Refresh token reuse detection**: Detect and block stolen tokens
* **Device fingerprinting**: Bind tokens to specific devices

## 🛡️ CSRF Protection Strategy

The project uses an **architectural approach to CSRF protection** rather than traditional CSRF tokens.

### How it works:

1. **Refresh tokens in httpOnly cookies** - Cannot be read by JavaScript
2. **Access tokens in Authorization headers** - Sent explicitly by client code
3. **CORS configuration** - Restricts origins

```js
app.use(cors({
    origin: 'http://localhost:5173',
    credentials: true,
    methods: ['GET', 'POST', 'PUT', 'DELETE', 'OPTIONS'],
    allowedHeaders: ['Content-Type', 'Authorization']
}));
```

### Why this mitigates CSRF:

* Attackers cannot read cookies due to `httpOnly`
* Attackers cannot set custom headers in simple requests
* Cross-origin requests require preflight (blocked by CORS)
* The API never accepts credentials via URL parameters or GET requests for state-changing operations

**Note**: This is sufficient for most SPA architectures, but for additional security, consider implementing **CSRF tokens** for state-changing operations or **SameSite cookie attributes** (already implemented with `sameSite: 'none'` in production).

## 🚨 Security Layers Summary

The Self Development Room implements **defense in depth**:

| Layer | Implementation | Purpose |
|-------|---------------|---------|
| **Authentication** | JWT Access + Refresh tokens | Identity verification |
| **Authorization** | Role-based middleware | Permission control |
| **Rate Limiting** | Redis-backed limiters | Brute force protection |
| **CSRF Protection** | httpOnly cookies + CORS | Cross-site attack prevention |
| **Password Hashing** | bcrypt with salt rounds | Credential storage security |
| **Token Expiration** | Short-lived access tokens | Limit exposure window |
| **Secure Cookies** | httpOnly, secure, sameSite | Cookie theft prevention |
| **Domain Separation** | Separate admin/user routes | Privilege isolation |

### What's Missing (Future Enhancements):

* ✅ **Token rotation**: Refresh tokens should rotate on each use
* ✅ **Token blacklisting**: Revoked tokens should be tracked in Redis
* ✅ **User-based rate limiting**: Protect against credential stuffing
* ✅ **Account lockout**: Temporary lockout after repeated failures
* ✅ **Audit logging**: Track authentication events
* ✅ **2FA/MFA**: Multi-factor authentication for sensitive operations

## 🧩 Mental Model Behind the Routes

The guiding question for the entire design was:

> **"Who are you, what can you do, and for how long?"**

* **Who** → authentication middleware (`authUser`, `authAdmin`)
* **What** → specific route and authorization logic
* **For how long** → token expiration (15min access, 7 days refresh)

This mental model ensures every request is:
1. **Identified** (who is making the request)
2. **Authorized** (what they're allowed to do)
3. **Time-bound** (for how long the permission is valid)

## 🏁 Conclusion

The routes of **Self Development Room** were designed with intention and security in mind.

They follow:

* Strong security principles
* Clear domain separation
* Scalable architecture
* Simple and readable structure

### Key Takeaways:

✅ **Separation of concerns** - Admin and user routes are completely isolated
✅ **Layered security** - Multiple defenses work together
✅ **Rate limiting** - Protects against brute force and abuse
✅ **Token-based auth** - Modern, stateless authentication
✅ **httpOnly cookies** - XSS-resistant refresh token storage

### Honest Assessment:

This architecture provides a **solid foundation**, but is **not production-complete** without:
* Token rotation and revocation
* User-based rate limiting
* Comprehensive audit logging
* Additional CSRF protections for sensitive operations

The design allows the project to grow **without painful refactors**, while maintaining full control over authentication, authorization, and data flow. As the application matures, adding the missing security layers will be straightforward thanks to the clean separation of concerns.
